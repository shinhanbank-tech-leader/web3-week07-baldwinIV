## 레포트 내용

주제 : 스마트컨트랙트 업그레이드 방식
 
1. 스마트 컨트랙트가 업데이트 불가능한 이유

블록체인 원장상 배포된 모든 코드들은 
  - 특정 관리지가 임의로 로직을 변경할 수 없게 하기 위해 (탈중앙화..!)
  - 모든 블록체인 노드들이 동일한 in에 같은 out을 내야하기 때문에
  - (기술적 : evm에서 컨트랙트 배포는 새 ca를 만들어 코드를 저장하는 행위이고, READ ONLY이기때문)
  의 사유로 해당 주소지에 대한 코드 수정이 정상적인 방식으로는 불가하다. 

2. 어떻게 하면 업그레이드가 가능할까

  - 상태 변수(데이터)를 저장하는 컨트랙트와 연산을 수행하는 컨트랙트를 따로 배포, 관리한다. 로직을 바꾸고 싶을 때 새로운 로직 컨트랙트를 배포한 뒤, 데이터 컨트랙트가 바라보는 로직 주소만 업데이트하는 방식이다.  (로직, 데이터 분리)
   -  사용자는 항상 고정된 주소(프록시)와 통신하고, 프록시가 실제 실행될 로직 컨트랙트로 호출을 꺾어주는 방식 (delegate call)

3. delegate call?
  일반적인 call은 A가 B를 호출하면, B의 코드가 B의 저장공간을 수정, msg.sender는 A
  delegate calld으로 A가 B를 호출하면, B의코드로 A의 저장공간을 수정, msg.sender는 A이고, 상태변화도 A의 스토리지에 기록

4. 스토리지 충돌?
 - EVM은 변수 이름이 아닌 슬롯번호(선언된 순서) 를 기준으로 데이터를 저장하기 때문에, 로직과 프록시에서 스토리지 레이아웃을 마치 우리 전문 구조체 관리하듯이 웜격하게 관리해야한다. 한 프록시에 여러로직이 생기기 시작하면 관리가 더욱 힘들어지니.. EIP-1967같은 표준도 존재한다.

 5. 은행 시스템과의 비교
    프록시 컨트랙트에느 implementation주소를 변경할수 있는 권한을 특정 주소로 부여할 있다. multisig 혹은 순차적 결재단계를 통한 주소변경 함수를 구현하면 은행 결재라인과 비슷하게 권한부여도 가능하다.
    프록시시스템을 통해 은행과 유사하게 이행된 코드에 대한 롤백도 가능할 것이다.
    다만, 잘못된 코드로 인해 잘못 기록된 데이터들은 되돌릴 수 없다. (불변)

 6. 결론 - 나의 생각

    - 은행은 데이터와 로직 모두 롤백이 가능하다.
    - 스마트컨트랙트는 로직은 프록시 등을 이용해 롤백이 가능하지만 변경된 데이터는 불변하다.
    - 시스템의 완전한 통제자로서 데이터와 로직 두개에 대한 통제가 불가하다.
    - 이것이 스마트컨트랙트와 asis시스템의 가장 큰 차이이다.

    계속 수업시간에 이야기가 나오는 것처럼.. 은행 코어.. 특히 거래가 많고 빠른 수신거래(수신을 잘 모르지만..) 적용하기엔 득보다는 실이 많아보인다.
    거래량이 상대적으로 적으나 묵직한 소유권 관련(RWA의 토큰화) 거래에 앞서 적용하는 것이 현실적으로 보인다.
